# SQLAlchemy Skill

> Think of SQLAlchemy models as ActiveRecord (Rails) models — they map to DB tables and support associations, validations (via constraints), and queries. Alembic = Rails migrations.

## Model Definition (ORM, Declarative Style)

```python
from sqlalchemy import String, Integer, ForeignKey, func
from sqlalchemy.orm import DeclarativeBase, Mapped, mapped_column, relationship
from datetime import datetime

class Base(DeclarativeBase):
    pass

class User(Base):
    __tablename__ = "users"

    id: Mapped[int] = mapped_column(primary_key=True)
    name: Mapped[str] = mapped_column(String(100), nullable=False)
    email: Mapped[str] = mapped_column(String(255), unique=True, nullable=False, index=True)
    deleted_at: Mapped[datetime | None] = mapped_column(default=None)  # Soft delete

    posts: Mapped[list["Post"]] = relationship("Post", back_populates="user")
```

- Use **`Mapped[]` type annotations** (SQLAlchemy 2.0 style) — they are type-safe and IDE-friendly.
- Always add **indexes** on columns used in WHERE / JOIN / ORDER BY clauses.
- Use **soft delete** (`deleted_at` column) instead of hard deletes — never use `DELETE` for user-facing records.
- Read the existing DB schema / migrations before writing new models to avoid mismatches.

## Session Management

```python
# FastAPI pattern (with dependency injection)
from sqlalchemy.orm import Session

def get_db():
    with SessionLocal() as db:
        yield db
```

- Always use context managers or `try/finally` to close sessions.
- Never share a session across requests.
- Use `AsyncSession` + `async_sessionmaker` for async apps.

## Querying

```python
# Fetch one or 404
from sqlalchemy import select

stmt = select(User).where(User.id == user_id, User.deleted_at.is_(None))
user = db.execute(stmt).scalar_one_or_none()

# Fetch many with eager loading (avoids N+1)
stmt = select(User).options(selectinload(User.posts)).where(User.deleted_at.is_(None))
users = db.execute(stmt).scalars().all()
```

- Always **filter out soft-deleted records**: `.where(Model.deleted_at.is_(None))`.
- **Avoid N+1 queries**: use `selectinload()` or `joinedload()` for related data (like Rails `includes`).
- Prefer `select()` (2.0 style) over `session.query()` (1.x legacy style) in new code.
- Add **indexes** to all columns used in filters and joins.

## Relationships

```python
# One-to-many
posts: Mapped[list["Post"]] = relationship("Post", back_populates="user", lazy="select")

# Many-to-many
tags: Mapped[list["Tag"]] = relationship("Tag", secondary="post_tags", back_populates="posts")
```

- Use `lazy="select"` (default) for most cases; use `selectinload` explicitly in queries when N+1 is a risk.
- Avoid `lazy="dynamic"` (deprecated in 2.0).

## Migrations (Alembic)

```bash
# Generate a new migration (like rails generate migration)
alembic revision --autogenerate -m "add_email_index_to_users"

# Apply migrations (like rails db:migrate)
alembic upgrade head

# Rollback one step (like rails db:rollback)
alembic downgrade -1
```

- Always review autogenerated migrations before applying — Alembic may miss complex changes.
- Keep `downgrade()` implemented for every migration to allow rollback.
- Schema changes require team lead approval (same as Rails guideline).

## Soft Delete Pattern

```python
# Filter active records (reusable)
def active(query):
    return query.where(Model.deleted_at.is_(None))

# Soft delete (never use db.delete())
def soft_delete(db: Session, instance) -> None:
    instance.deleted_at = datetime.utcnow()
    db.commit()
```

## Testing

- Use a separate test database; wrap each test in a transaction and roll back after.
- Use `pytest` fixtures for session and test data setup.
- Test that soft-deleted records are excluded from queries.
- Target **≥95% coverage** on repository/service methods that touch the DB.
